# Текущее состояние Hobbes

Дата актуализации этого документа:

- `2026-03-18`

Этот документ нужен как короткая каноническая сводка.
Он не заменяет rollout-документы и changelog, а собирает главное в одном месте.

## 1. Что уже реально работает

### Базовый живой контур

Рабочий baseline сейчас такой:

- Telegram DM является основным рабочим пользовательским входом;
- OpenClaw на VPS живой и обслуживает gateway / health / agent workspaces;
- `main` является рабочим фронтовым агентом;
- control-layer агенты `chief`, `comms`, `guard` существуют и используются;
- Phase 3 specialists `research`, `memorykeeper`, `bookingprep` развернуты и доступны через маршрутизацию;
- dashboard на Railway получает live snapshots и показывает overview;
- Control Center умеет читать allowlisted-файлы, писать в GitHub и синхронизировать часть runtime-файлов на VPS.

### Search и routing

Важный текущий шаг зрелости:

- у Hobbes есть category-aware search router;
- `chief` выступает как точка маршрутизации поисковых задач;
- `research` владеет source-backed поиском;
- `bookingprep` выделен под travel/booking-тип задач;
- поисковые события частично видны в dashboard через `Searches`.

## 2. Что работает частично

### Telegram groups

Группы уже не находятся в состоянии "совсем не подключено":

- policy-компиляция для групп есть;
- behavior profiles есть;
- allowlist-подход есть;
- runtime artifacts для групп генерируются.

Но это еще не означает production-ready групповой режим.
По-настоящему надежный массовый group rollout пока не доказан.

### Reminders

Напоминания уже хорошо нормализуются на уровне intake.

Но это пока еще не то же самое, что:

- реальный scheduler;
- гарантированная доставка;
- retries;
- прозрачное подтверждение успешной постановки.

### Dashboard observability

Dashboard уже полезен.
Но он пока не равен полноценной операционной консоли зрелой multi-agent платформы.

На сегодня:

- часть данных строится по snapshots;
- approvals еще не доведены как first-class operational events;
- usage/cost пока неполны;
- run chains пока во многом восстанавливаются по косвенным данным.

## 3. Что все еще слабое

### Search verticals

Самые слабые места на сегодня:

- `local_maps`
- `travel_booking`

Проблема уже не в том, что система "не умеет искать вообще".
Проблема в том, что некоторые классы retrieval-задач требуют более сильных backend-стратегий и структурированного подтверждения результата.

### Test harness

Один из самых болезненных operational gaps:

- локальная CLI-валидация внутренних агентов и routed flows все еще ненадежна.

Это мешает безопасно масштабировать agent topology, потому что каждый новый агент становится сложнее проверять до production.

### Memory governance

Контур памяти уже обозначен хорошо по роли.
Но как полноценно работающая система он еще слишком тонок.

Пока не хватает явной зрелости в таких местах, как:

- memory classes;
- ownership;
- lifecycle записей;
- auditability;
- clean write proposals vs durable write execution.

## 5. Практический вывод

Сегодня Hobbes уже можно описывать не как "одного Telegram-бота", а как раннюю, но настоящую multi-agent систему с dashboard-контуром.

Но это еще не стадия "добавляй сколько угодно новых агентов".

Правильнее сказать так:

- базовая агентная архитектура уже есть;
- наблюдаемость уже появилась;
- маршрутизация уже настоящая;
- а вот инфраструктура зрелой масштабируемости еще не доведена до конца.
