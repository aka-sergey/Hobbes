# Пробелы Hobbes и следующий порядок развития

Этот документ нужен как короткая инженерная правда:
не что "хотелось бы когда-нибудь", а какие пробелы сегодня реально мешают безопасно наращивать систему.

## 1. Главные пробелы

### 1. Нет полноценной событийной модели рантайма

Сейчас dashboard в значительной степени работает на snapshots и косвенных признаках.

Это означает:

- не все run chains видны как first-class события;
- approvals еще не живут как полноценный operational flow;
- usage и cost не собираются как зрелая телеметрия;
- часть правды о системе приходится восстанавливать по session-файлам и журналам.

Пока это не исправлено, масштабирование агентного слоя будет всегда хуже наблюдаться, чем хотелось бы.

### 2. Approval-layer есть концептуально, но еще не как полный workflow

`guard` уже работает как risk gate.
Но пока еще не собран полный machine-readable контур:

- запрос согласования;
- pending state;
- human resolution;
- execution after approval;
- trace в dashboard.

Это один из важнейших пробелов перед появлением более "мощных" агентов.

### 3. Search maturity неравномерная

Архитектура поиска уже стала лучше.
Но по качеству retrieval и подтверждаемости результата есть слабые зоны:

- `travel_booking`
- `local_maps`

Пока эти вертикали не усилены, нельзя честно считать Hobbes зрелым источником для serious real-world lookup в этих классах задач.

### 4. Reminder execution пока не доказан как production-feature

Сейчас reminders хорошо нормализуются как задача.
Но пока еще не доказан надежный delivery loop:

- scheduler;
- persisted storage;
- retry;
- execution confirmation.

То есть напоминания пока ближе к "умному intake", чем к надежной службе доставки.

### 5. Групповой Telegram-контур еще не прошел зрелую live-валидацию

Есть:

- allowlist;
- profiles;
- compiled prompts;
- group policy compilation.

Но нет еще такой уверенности, при которой можно безопасно сказать:
"Открываем Hobbes на реальные группы без оговорок."

### 6. Тестирование routed multi-agent цепочек все еще слишком хрупкое

Проблемы:

- локальный CLI harness ненадежен;
- synthetic проверки могут давать false negatives;
- часть validation по-прежнему опирается на session growth и косвенные признаки.

Это серьезный тормоз для роста системы.
Каждый новый агент увеличивает сложность именно там, где инструменты проверки пока слабее нужного уровня.

### 7. Memory layer пока обозначен сильнее, чем реализован

`memorykeeper` есть как роль.
Но для зрелой памяти нужно сильнее зафиксировать:

- классы памяти;
- права записи;
- дедупликацию;
- хранение источника факта;
- границу между staging и durable memory.

### 8. Source of truth уже понятен, но не везде достаточно формализован

Для части editable-файлов GitHub уже является каноническим источником.
Но для некоторых operational truth-слоев остается напряжение между:

- repo truth;
- runtime truth на VPS;
- session truth;
- dashboard truth.

Чем больше будет агентов, тем больнее станут такие расхождения.

## 2. Что это означает для новых агентов

На сегодня главный риск не в том, что "агентов мало".

Главный риск в другом:
если нарастить новых агентов раньше, чем дозреют observability, approvals, test harness и artifact discipline, система станет внешне богаче, но реально менее управляемой.

Проще говоря:

- новые агенты сейчас можно добавлять;
- но это нужно делать выборочно и только вместе с укреплением инфраструктуры.

## 3. Какой порядок развития сейчас выглядит самым здоровым

### Этап 1. Добить инфраструктуру агентности

Прежде всего нужны:

- structured run events;
- first-class approvals;
- usage / cost telemetry;
- artifact paths;
- стабильная routed validation;
- ясные runtime truth rules.

### Этап 2. Добить слабые вертикали

В первую очередь:

- `travel_booking`
- `local_maps`
- reminder execution
- Telegram group validation

### Этап 3. Только потом расширять агентную систему дальше

После этого уже можно более уверенно добавлять новых агентов.

Хорошие кандидаты на будущее:

- inbox/workflow agent;
- durable memory operations agent;
- stronger document-production layer;
- later device / node-host layers, но не раньше.

## 4. Практический критерий для нового агента

Новый агент стоит добавлять только если для него заранее понятны:

- ownership;
- входной тип задач;
- какой artifact он производит;
- как он проходит через `chief`;
- нужен ли для него `guard`;
- как его проверять локально и на VPS;
- как его видно в dashboard.

Если на эти пункты нет ответа, то вы пока придумываете не агента, а новую зону хаоса.

## 5. Самый важный вывод

Hobbes уже дорос до стадии, где дальнейший прогресс зависит не только от промптов и новых ролей.

Дальше качество системы будет определяться тем, насколько хорошо вы укрепите:

- наблюдаемость;
- событийную модель;
- approval workflow;
- качество тестирования;
- дисциплину артефактов;
- границы памяти.
