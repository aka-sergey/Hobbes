# Контрольная Панель Hobbes

Дата:

- `2026-03-18`

## Назначение

Этот документ объясняет, как пользоваться контрольной панелью Hobbes:

- как редактировать документы и политики
- чем отличаются действия `Сохранить черновик`, `Применить в GitHub`, `Sync на VPS`
- какие файлы можно менять безопасно
- как не допускать рассинхрона между GitHub и живым VPS

Панель доступна здесь:

- [Control Center](https://hobbes-dashboard-web-production.up.railway.app/control)

## Что это за панель

Контрольная панель — это не SSH и не файловый менеджер.

Это безопасный пульт управления для Hobbes, который позволяет:

- читать ключевые `.md` и `.json` файлы
- редактировать allowlisted файлы
- сохранять черновики
- коммитить изменения в GitHub
- синхронизировать часть runtime-файлов на VPS

## Как устроен интерфейс

### Левая колонка

Показывает список файлов по разделам:

- `Документы`
- `Политики`
- `Тестирование`
- `Агенты`

Смысл:

- `Документы` — справка, статус, архитектура, чекпойнты
- `Политики` — правила поведения Hobbes
- `Тестирование` — режимы и вопросники для проверки ботов
- `Агенты` — persona, reminders, document shapes и другие рабочие контракты агентов

### Центральная колонка

Это редактор выбранного файла.

Здесь вы:

- меняете текст
- правите JSON
- редактируете markdown

### Правая колонка

Это:

- preview
- статус файла
- источник версии
- краткая расшифровка для части конфигов

## Что означают кнопки

### `Сохранить черновик`

Что делает:

- сохраняет текущую редакторскую версию в базе dashboard
- не меняет GitHub
- не меняет VPS

Когда использовать:

- если вы хотите сохранить промежуточную версию
- если правки еще не готовы к публикации

### `Применить в GitHub`

Что делает:

- записывает текущую версию файла в репозиторий `aka-sergey/Hobbes`
- создает реальный commit в GitHub

Что не делает:

- не гарантирует автоматическое применение на VPS

Когда использовать:

- когда вы уверены, что файл уже надо зафиксировать в репозитории

### `Sync на VPS`

Что делает:

- берет allowlisted файл и докатывает его на живой сервер
- если файл runtime-зависимый, Hobbes начинает использовать новую версию на VPS

Примеры:

- `config/telegram/chat_policies.example.json`
- `config/agents/main/workspace/PERSONAS.md`
- `config/agents/comms/workspace/PERSONAS.md`

Что важно:

- работает только для файлов с типом `repo_and_runtime`
- для Telegram chat policies дополнительно запускается компиляция runtime-конфига

### `Сбросить к исходнику`

Что делает:

- убирает несохраненные локальные правки в редакторе
- возвращает текст к текущей исходной версии файла

Что не делает:

- не откатывает GitHub
- не откатывает VPS

## Главный принцип работы

Правильный порядок действий:

1. Открыть нужный файл
2. Внести правки
3. Нажать `Сохранить черновик`, если хотите сохранить промежуточную версию
4. Нажать `Применить в GitHub`
5. Если это runtime-файл — нажать `Sync на VPS`
6. Проверить поведение Hobbes в Telegram или через dashboard

## Какие файлы можно менять безопасно

### `Документы`

Это безопасные файлы описания и статуса:

- Telegram current state
- search current state
- architecture docs
- guides

Изменение этих файлов не меняет поведение бота напрямую.

### `Политики`

Это уже рабочие настройки:

- `config/telegram/chat_policies.example.json`
- `config/telegram/test_mode.example.json`

Именно они влияют на:

- в каких чатах бот работает
- как он включается
- когда молчит
- какую persona использует

### `Агенты`

Это рабочие контракты поведения:

- `PERSONAS.md`
- `REMINDERS.md`
- `MEETING_PREP.md`
- `DOCUMENT_SHAPES.md`

Они влияют на:

- стиль ответа
- правила поведения
- форматы выдачи
- то, как агенты упаковывают результат

## Какие изменения влияют только на репозиторий

Файлы типа `repo_only`:

- обновляются в GitHub
- полезны для документации и source-of-truth
- но не должны считаться автоматически примененными в live runtime

## Какие изменения влияют на живого бота

Файлы типа `repo_and_runtime`:

- после `Применить в GitHub` меняют исходник
- после `Sync на VPS` начинают влиять на живого Hobbes

Важно:

- если вы сделали только `Применить в GitHub`, но не сделали `Sync на VPS`, live behavior может остаться старым

## Как не допустить рассинхрон

Вот главный рабочий регламент:

1. Основной способ изменений — через `/control`
2. `Применить в GitHub` — всегда перед `Sync на VPS`
3. Ручной SSH использовать только для аварийных hotfix
4. После ручного hotfix желательно вернуть изменения обратно в репозиторий

Почему это важно:

- иначе GitHub, локальный repo и VPS начинают жить разными версиями

## Как понять, что панель работает

Признаки, что все в порядке:

- после `Применить в GitHub` появляется commit в истории
- после `Sync на VPS` поведение Hobbes реально меняется
- runtime-файл обновляется на сервере

## Что уже точно работает

По текущему состоянию:

- `Применить в GitHub` реально создает коммиты
- `Sync на VPS` реально обновляет runtime-файлы
- Telegram group policy уже компилируется из chat policies в live runtime

## Где still есть осторожность

Панель рабочая, но важно помнить:

- если одновременно менять одно и то же через `/control`, git и SSH, можно запутать source of truth
- runtime и repo должны обновляться последовательно, а не хаотично

## Рекомендуемый способ работы

Для большинства задач:

1. Меняете файл в `/control`
2. Смотрите preview
3. Нажимаете `Применить в GitHub`
4. Для runtime-файла жмете `Sync на VPS`
5. Делаете один тест

Для аварий:

1. Hotfix по SSH
2. Потом обязательно зафиксировать это же изменение в репо
